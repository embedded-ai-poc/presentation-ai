Agent 시대의 업무 방식

나: "로그 파싱 모듈 리팩토링해줘. 기존 테스트 유지하고, 새 기능 테스트도 추가해." AI: 파일 분석 중... → 기존 테스트 확인 → 리팩토링 진행 → 새 테스트 작성 → 전체 테스트 실행 (32/32 pass) → 결과 보고 나: (diff 검토 후 승인)

Pilot

가장 중요한 요소 = 당신

Pilot의 핵심 역량

  • 문제 분해 — 복잡한 문제를 AI가 처리 가능한 단위로
  • 실패 패턴 인식 — 할루시네이션, 컨텍스트 한계 예측
  • 도메인 지식 — AI 출력의 적절성을 즉시 판단
  • 파이프라인 설계 — 반복 가능한 워크플로우 구축

문제 분해 예시

3단계: 장바구니
각 단계 검증 후 진행

세 가지가 맞으면

Model이 좋아지든, Harness가 발전하든

첫 대화 시작하기

# 프로젝트 폴더로 이동 cd ~/my-project # Claude Code 시작 claude # 첫 번째 요청 > 이 프로젝트 구조 설명해줘

"이 함수 설명해줘"

> gc_manager.c의 collect_garbage 함수가 뭘 하는 건지 설명해줘 AI: 이 함수는 SSD의 Garbage Collection을 담당합니다. 1. 먼저 valid page 비율이 가장 낮은 블록을 선정... 2. 해당 블록의 valid page들을 새 블록으로 복사... 3. 원본 블록을 erase... ...

코드를 읽어주고 분석해줍니다

"좀 더 쉽게 설명해줘"

> 12살한테 설명하듯이 다시 설명해줘 AI: SSD는 데이터를 저장할 때 마치 책장에 책을 꽂는 것처럼 해요. 그런데 책을 지우면 빈 공간이 생기죠? GC는 청소부 아저씨예요. - 빈 공간이 많은 선반을 찾아서 - 남아있는 책들을 다른 선반으로 옮기고 - 선반을 깨끗이 비워요 그러면 새 책을 넣을 공간이 생기죠!

모르는 것 계속 파고들기

> 왜 valid page 비율이 낮은 블록을 선택하는 거야? > "erase"는 뭐야? 왜 필요해? > NAND Flash의 특성상 erase가 필요한 이유가 뭐야? > 그럼 erase 횟수에 제한이 있어?

Agent 맛보기

"시켜놓고 지켜보기"의 진정한 맛

"분석하고 테스트 추가해줘"

> wear_leveling.c 분석하고 테스트 케이스 추가해줘 AI: 파일 분석 중... → 현재 테스트 커버리지 확인 → 누락된 엣지 케이스 식별 → test_wear_leveling.c에 테스트 추가 중 → 빌드 확인... → 테스트 실행: 5/5 pass 완료! 다음 테스트 케이스를 추가했습니다: - test_block_erase_count_limit - test_wear_leveling_threshold ...

시켜놓고 지켜보기

AI 작업 중 (2-3분) 관찰하면서 질문하기
"왜 이 파일을 먼저?" "왜 여기서 막혔지?"

관찰할 것들

  • 어떤 파일을 먼저 읽는가 — AI의 탐색 전략
  • 어디서 헤매는가 — 컨텍스트 부족 신호
  • 어떤 명령을 실행하는가 — 작업 방식 이해
  • 실패했을 때 어떻게 복구하는가 — 에러 핸들링

실습: littlefs 탐색

# littlefs 클론 git clone https://github.com/littlefs-project/littlefs.git cd littlefs # Claude Code로 탐색 claude > 이 프로젝트 구조 설명해줘 > lfs_file_write 함수가 어떻게 동작하는지 설명해줘 > 이 함수에 대한 테스트 케이스 추가해줘

결과 검토하기

# diff 확인 git diff # 변경 사항 검토 > 왜 이렇게 구현했어? > 더 나은 방법은 없어? # 만족스러우면 승인 git add . git commit -m "Add test cases for lfs_file_write"
