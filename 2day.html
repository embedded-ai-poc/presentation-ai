왜 파이프라인인가

  • 반복 가능 — 매번 같은 품질 보장
  • 검증 포인트 — 각 단계에서 확인
  • 실패 시 복구 — 어디서 문제인지 명확
  • 자동화 가능 — 워크플로우로 패키징

AI에게 역할 부여하기

"너는 시니어 요구사항 분석가야. 스펙의 핵심을 정리해줘." "너는 PRD 작성 전문가야. 요구사항 기반으로 문서 작성해줘." "너는 TDD의 달인이야. 테스트 먼저 작성하고 구현해줘." "너는 코드리뷰의 화신이야. MISRA C 기준으로 검토해줘."

TDD 프롬프트 패턴

# Step 1: 테스트 먼저 (Red) "wear_leveling_select_victim 함수의 테스트 케이스를 먼저 작성해줘. 구현은 하지 마." # Step 2: 최소 구현 (Green) "이 테스트들이 통과하도록 최소한으로 구현해줘." # Step 3: 리팩토링 (Refactor) "테스트는 유지하면서 코드 품질을 개선해줘."

TDD 실습 예시

# Step 1: 테스트 먼저 작성 (Red) > wear_leveling_select_victim 함수에 대한 테스트 케이스 먼저 작성해줘. - 정상 케이스: erase count가 가장 낮은 블록 선택 - 엣지 케이스: 모든 블록 erase count 동일 - 에러 케이스: 선택 가능한 블록 없음 # Step 2: 최소 구현 (Green) > 이 테스트들이 통과하도록 최소한으로 구현해줘. # Step 3: 리팩토링 (Refactor) > 코드 품질 개선해줘. 가독성, 성능 위주로.

# CLAUDE.md에 추가 ## 테스트 요구사항 - 단위 테스트 커버리지: 80% 이상 - 브랜치 커버리지: 70% 이상 - 에러 핸들링 경로: 100% # 프롬프트 "이 함수의 테스트 커버리지가 80%가 되도록 테스트 케이스를 추가해줘."

실습 시간

담당 모듈에서 함수 하나 선택해서 TDD

실습 단계

  • Step 1 — 테스트할 함수 선정 (담당 모듈에서)
  • Step 2 — "테스트 케이스 먼저 작성해줘" 요청
  • Step 3 — 테스트 실행 → 실패 확인 (Red)
  • Step 4 — "테스트 통과하도록 구현해줘" 요청
  • Step 5 — 테스트 실행 → 성공 확인 (Green)
  • Step 6 — "리팩토링해줘" 요청 (Refactor)
